
# UE5-Survival-Kit (UE5.7.4 / Linux / Shared-Server Edition)

**「権限のあるホーム領域は24GBしかないが、お化けスペックのCPU/GPUと、86TBのNASがある」**

本リポジトリは、そんな極限の限界環境で **Unreal Engine 5.7.4** を無理やり叩き起こすための「生存戦略（思想）」とそのプロトタイプスクリプトです。
（※本スクリプトは UE5.7.4 をターゲットに最適化されています）

## 核心にある思想 (The Philosophy)
公式のドキュメントは、潤沢なローカルストレージがある「理想的な環境」を前提としています。しかし、現実の共有サーバーや制約のある環境はそうではありません。

本キットの思想は**「計算資源と保存領域の完全な分離（Decoupling）」**にあります。
- **計算（CPU/GPU/メモリ）**: 高性能なローカルサーバーに使い捨てのDockerコンテナとして召喚。
- **保存（ソース・キャッシュ・成果物）**: 枯渇寸前のローカルSSDを避け、広大なNAS領域へすべて流し込む。
- Dockerのマウント（`-v`）をハックし、NASをコンテナの「ホーム領域」と錯覚させることで、物理的な容量制限を論理的にバイパスします。

## 📋 前提条件 (Prerequisites)
本スクリプトは、以下のインフラ構成であることを前提として設計されています。
* **直接接続されたデスクトップ環境**: 実際に操作・画面表示を行うデスクトップPCが、計算用サーバー（Linux/Ubuntu）に直接接続、あるいは超高速なローカルネットワークで帯域が確保されていること。（サーバー側で `-nullrhi` 等の軽量設定で起動したエディタを、デスクトップ側から透過的に叩く、あるいは表示を受け取るための前提です）
* **極小のホーム領域と巨大なNAS**: ローカルの権限がある領域（Homeなど）が数十GBレベルで枯渇しており、データ保存先として大容量の外部ストレージ（NAS等）がマウントされていること。

## 使い方 (Usage)
1. 本リポジトリを適当な場所にクローンする。
2. `build_ue5.sh` 内の先頭にある環境変数（`NAS_PATH` 等）を、自身の環境に合わせて書き換える。
3. スクリプトに実行権限を付与して起動する。

```bash
chmod +x build_ue5.sh
./build_ue5.sh
